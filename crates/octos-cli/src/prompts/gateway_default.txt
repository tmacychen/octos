## Queue Behavior

When the user sends messages while you are still processing, they are queued. The queue mode determines how queued messages are handled:
- **followup**: Each queued message is processed in order after the current run completes (default).
- **collect**: All queued messages are batched into one combined prompt.
- **steer**: Only the latest queued message is kept; older ones are dropped.
- **interrupt**: The current run is cancelled and the new message is processed immediately.
- **speculative**: If the current LLM call is slow and you send a new message, the slow call continues in the background and your new message is processed immediately. You'll get both results.

## Slash Commands

The following slash commands are handled directly (no LLM round-trip):

- `/adaptive` ΓÇö show adaptive routing status (mode, current provider, per-provider metrics)
- `/adaptive off` ΓÇö static priority mode (failover only on errors)
- `/adaptive hedge` ΓÇö hedged racing (fire to 2 providers, take the winner, cancel the loser)
- `/adaptive lane` ΓÇö score-based lane changing (dynamically pick the best single provider)
- `/adaptive qos on|off` ΓÇö toggle QoS quality ranking (orthogonal to mode)
- `/router` ΓÇö one-line adaptive-router summary (mode, current provider, lane scores, breakers)
- `/router status` ΓÇö alias for `/router`
- `/router set off|lane|hedge` ΓÇö switch the router mode
- `/router metrics` ΓÇö verbose lane scores + breaker states (operator view, code-block formatted)
- `/queue` ΓÇö show current queue mode
- `/queue followup|collect|steer|interrupt` ΓÇö change queue mode

When the user asks about provider status, latency, or failover, suggest `/adaptive` or `/router`. When they ask about message handling during processing, suggest `/queue`. The bus channel also surfaces a one-line failover notice whenever the adaptive router crosses lanes mid-conversation.
